## 1. Введение
Наименование программы или программного изделия: «EventHub» (Платформа для организации и поиска мероприятий).
EventHub — это клиент-серверное веб-приложение, предназначенное для создания, продвижения и поиска событий (мероприятий). Платформа позволяет организаторам публиковать информацию о событиях, управлять билетами, а пользователям — находить интересные мероприятия, бронировать билеты и взаимодействовать в рамках событийного сообщества. Разработка ведется в учебных целях для демонстрации навыков полноценной разработки.

## 2. Основание для разработки
Разработка ведется в рамках учебного (курсового) проекта для портфолио. Основанием является задание на разработку клиент-серверного приложения с использованием стека технологий: фронтенд (HTML, CSS, JavaScript) и бэкенд (C++) с базой данных PostgreSQL.

## 3. Назначение разработки
Функциональное назначение: Создание централизованной цифровой платформы, которая упрощает процесс организации мероприятий и взаимодействия между их участниками.
Эксплуатационное назначение: Приложение предназначено для использования организаторами мероприятий и конечными пользователями через веб-браузер. Система обеспечивает создание событий, управление регистрацией, поиск и бронирование.

## 4. Требования к программе или программному изделию
**4.1. Требования к функциональным характеристикам**

**Публичная часть (фронтенд):**
* Главная страница с описанием сервиса и призывом к действию.

* Страница «Каталог мероприятий» с карточками событий, фильтрацией по дате, категории, местоположению.

* Страница «О проекте» с информацией о команде и ценностях.

* Система регистрации и аутентификации пользователей (организатор/участник).

**Личный кабинет пользователя (фронтенд + бэкенд):**

* Просмотр и редактирование профиля.

* Раздел «Мои мероприятия» (предстоящие и архив).

* Раздел «Мои билеты» с отображением статуса и QR-кодами.

**Личный кабинет организатора (фронтенд + бэкенд):**

* Форма для создания и редактирования мероприятий с указанием названия, описания, даты, места, типа билетов.

* Просмотр списка созданных мероприятий и статистики по ним (количество зарегистрированных участников).

**Административный функционал (бэкенд):**

* Модерация создаваемых мероприятий.

* Управление категориями мероприятий.

**Уникальные функциональные возможности (фичи):**

* Живая лента активности: Динамическое отображение действий пользователей в реальном времени (например, «Анна присоединилась к событию X»).

* Система рекомендаций: Персонализированная выдача мероприятий на основе истории просмотров и предпочтений пользователя.

* Интерактивная карта совместимости: Визуализация в профиле пользователя, показывающая процент совпадения интересов с другими участниками предстоящего события.

**4.2. Требования к надежности**

* Гарантированная целостность и сохранность пользовательских данных, информации о мероприятиях и транзакциях.

* Обработка ошибок ввода данных и сетевых сбоев с информативными сообщениями для пользователя.

* Резервное копирование критических данных базы данных (например, списка пользователей и мероприятий).

**4.3. Условия эксплуатации**
* Приложение должно корректно работать в последних версиях современных браузеров (Google Chrome, Mozilla Firefox, Microsoft Edge).

* Интерфейс должен быть адаптивным и обеспечивать удобное использование как на компьютерах, так и на мобильных устройствах.

* Работа в среде с постоянным доступом в Интернет.

**4.4. Требования к составу и параметрам технических средств**

**Серверная часть (бэкенд):**

* Язык программирования: C++ (стандарт 17 и выше).

* HTTP-сервер/библиотека: cpp-httplib или Boost.Beast.

* База данных: PostgreSQL 12+.

* Клиент для работы с БД: libpqxx.

* ОС для разработки и развертывания: Windows / Linux.

**Клиентская часть (фронтенд):**

* Языки разметки и программирования: HTML5, CSS3, JavaScript (ES6+).

* Библиотеки/фреймворки: чистый JS или Vue.js/React (на усмотрение разработчика) для динамических элементов.

**4.5. Требования к информационной и программной совместимости**
* Обмен данными между клиентом и сервером в формате JSON.

* Использование REST-like API для взаимодействия фронтенда и бэкенда.

* Фронтенд должен быть собран и развернут как статический набор файлов.

* Код должен быть написан с учетом переносимости и возможности сборки с помощью системы CMake.

## 6. Технико-экономические показатели
Разработка является учебным проектом. Основные показатели:

* Ожидаемый результат: Готовое к демонстрации клиент-серверное приложение, которое можно представить в портфолио как пример полноценной разработки.

* Экономические преимущества: Нулевая стоимость разработки, использование свободного программного обеспечения и инструментов.

* Ценность: Приобретение практических навыков в full-stack разработке, работе с C++ в веб-среде, проектировании баз данных и создании современного пользовательского интерфейса.

## 7. Стадии и этапы разработки
1. Стадия 1. Подготовительный этап (до 20.10):

* Выбор и утверждение темы проекта.

* Формирование ТЗ.

* Проектирование архитектуры БД и API.

2. Стадия 2. Разработка бэкенда (до 15.11):

* Настройка сервера на C++ (cpp-httplib/Boost.Asio).

* Подключение к PostgreSQL, реализация основных моделей (Пользователь, Мероприятие, Билет).

* Реализация базовых endpoints API (CRUD для мероприятий, регистрация пользователей).

3. Стадия 3. Разработка фронтенда (до 05.12):

* Верстка основных страниц (Главная, Каталог, О проекте) на основе предоставленных CSS.

* Реализация интерактивных элементов на JavaScript.

* Интеграция фронтенда с API бэкенда (отправка запросов, обработка ответов).

4. Стадия 4. Интеграция и тестирование (до 20.12):

* Связывание всех модулей в единую систему.

* Функциональное тестирование.

* Наполнение тестовыми данными.

* Финальная отладка и подготовка к демонстрации.

## 8. Порядок контроля и приемки

**Контроль: Выполнение каждого этапа проверяется по наличию соответствующего функционала и закоммиченного кода в репозиторий GitHub/GitLab.**

**Приемка (Критерии готовности):**

1. Приложение успешно собирается и запускается локально.

2. Реализованы все основные страницы интерфейса.

3. Работает регистрация и аутентификация пользователей.

4. Организатор может создать мероприятие и увидеть его в каталоге.

5. Пользователь может найти мероприятие и «забронировать» билет (создать запись в БД).

6. Данные между фронтендом и бэкендом передаются корректно.

* Результат приемки: Рабочее приложение, размещенное в публичном репозитории, с подробным README.md.
